**WHAT AND WHY:**

The project is an EEG-driven expressive presence system for VR (Resonite, Quest 3). The core idea comes from Jaron Lanier's concept of post-symbolic communication — specifically his observation that cephalopods (octopi, squid) communicate by changing the pigmentation, texture, and shape of their entire skin in real time, expressing internal state directly as a visible surface rather than through symbolic language. Lanier argued VR could enable a similar mode of communication for humans: instead of saying something, you *become* it. Your internal state shapes the world around you or the surface of your avatar directly.

This project is a practical attempt at that. It is **not** a control system. It is not about moving objects with your mind or issuing commands. It is about making brain state continuously visible as a living, reactive skin on an avatar or object in Resonite — an expressive coat that breathes and shifts with your internal state. Think: chromatophores. The EEG is treated as a noisy, expressive signal source, not a precise input device.

This aligns with Josh's broader artistic practice around generative art, non-verbal expression, and exploring AI/BCI from an artist's perspective (he previously made a generative HTML canvas piece called "emergence" and a YTP-style video exploring what it's like to be an LLM).

**TGAM DATA — EVERYTHING AVAILABLE:**

The Arduino Brain Library CSV outputs these values once per second:

| Field | Range | Notes |
|---|---|---|
| signal_quality | 0–200 | 0 = perfect, 200 = no contact. Gate all output on this. |
| attention | 0–100 | NeuroSky eSense — proprietary algorithm |
| meditation | 0–100 | NeuroSky eSense — proprietary algorithm |
| blink | 0–255 | Event-based, fires on detected blink. Separate packet, may not appear every cycle. |
| delta | raw int | Deep sleep, stillness, large values |
| theta | raw int | Creative flow, drowsiness, meditation |
| low_alpha | raw int | Relaxed awareness |
| high_alpha | raw int | Eyes-closed relaxation |
| low_beta | raw int | Active thinking, concentration |
| high_beta | raw int | Stress, intense focus, anxiety |
| low_gamma | raw int | Perceptual binding |
| high_gamma | raw int | High cognitive load |

**Important:** Raw band integers are enormous (often 50,000–500,000+) and vary wildly between people and sessions. They must be normalized before use. The blink value is also available as a separate detection event with strength 0–255.

No heart rate, no breath rate, no motion data — the TGAM only outputs the above. Those would require additional hardware which is out of scope for this phase.

**WHAT WAS PLANNED AND WHY:**

**Send ALL bands to Resonite** — not just attention/meditation. The community approach (BrainFlowsIntoVRChat / BrainFlowsIntoVR) confirms this is the right call. All 8 band power values + attention + meditation + blink + derived values all get sent as separate OSC float addresses. Resonite can ignore what it doesn't need. The bands are where the expressive richness lives — attention/meditation alone are too coarse.

**EMA smoothing** — Exponential Moving Average, standard in the VRChat BCI community. Weights recent samples more heavily than a rolling average, so it responds to genuine changes while smoothing noise. The 1 packet/second update rate from the TGAM would cause jarring snapping without this. Different alpha values per signal type: bands 0.2, attention/meditation 0.15 (slower/more organic), blink 0.5 (faster, event-like), derived 0.1 (very smooth).

**Session min/max normalization** — Raw band values vary hugely between people and sessions. Min/max scaling tracks the running min and max per band during the session and maps everything to 0.0–1.0 within observed range. Self-calibrates automatically — no manual setup needed. Has a 30-second warmup period while the range establishes, marked with a "CALIBRATING" flag in the UI and a `/eeg/system/warmup_done` OSC address.

**Derived values** — Computed on the Python side before hitting Resonite, more expressive than raw bands:
- `alpha_beta_ratio` — classic relaxation vs activation indicator
- `engagement_index` — beta / (alpha + theta), focus measure
- `theta_alpha_ratio` — creative/flow state indicator  
- `band_energy` — overall neural "volume", sum of all normalized bands
- `band_dominance` — which band is currently highest (0.0–7.0 float, maps to band index)
- `dominant_band` — string name of dominant band (for logging/ML)

**Signal quality gating** — When signal_quality > threshold (default 50), OSC output is frozen at last known good values and a `/eeg/system/signal_ok = 0.0` flag is sent. This prevents garbage data from corrupting the visual when the electrode loses contact during movement.

**Session CSV logging** — Built in from day one specifically to support the future ML layer. Every session writes a timestamped CSV with all raw, processed, and derived values. This is the training data source for intent classification later.

**Modular pipeline architecture** — Designed so the ML classification module slots in later as an optional add-on without touching the base system:
```
Serial Reader → Processor (EMA + normalize + derive) → OSC Sender → Resonite
                                                     → Logger (CSV)
                                                     ↓ (future)
                                              ML Classifier → /eeg/class/intent OSC
```

**FUTURE ML LAYER (not built yet, planned):**

The Instagram inspiration: someone doing calibrated mental imagery classification. The user speaks a phrase/word aloud while EEG is recorded (N repetitions), building a template of what brain activity looks like during that phrase. During use, incoming EEG windows are compared against templates and the nearest match fires. It's not real mind-reading — it exploits the fact that vocalized and imagined speech share some frontal EEG patterns, especially subvocalization activating motor planning circuits.

Plan: sklearn KNN or cosine similarity on band feature vectors. Small vocabulary (3–5 classes max with single-channel TGAM). Optional, session-calibrated, completely separate from base pipeline. OSC address: `/eeg/class/intent` as string label or int index. The session logging was specifically designed to capture training data for this.

**RESONITE INTEGRATION PLAN:**

Resonite has built-in OSC support. The signal chain in Resonite for each value:
```
OSC_Receiver component
  → OSC_Value (float) component at matching address
    → Value Smooth Lerp<float> node in ProtoFlux (smooths the 1Hz update on Resonite side)
      → Value Field Drive<float> node
        → material parameter / blendshape / anything else
```

**The skin approach — material stacking:**
Multiple semi-transparent mesh renderers stacked on the same object (avatar surface or separate object), each layer driven by a different EEG band. Like biological chromatophore layers:
- Layer 1 (base/opacity): driven by meditation — how "present" or opaque the surface is
- Layer 2 (color/pattern): driven by alpha/theta — warm vs cool, organic shifting
- Layer 3 (accent/detail): driven by beta/attention — sharp details activate with focus
- Layer 4 (event flash): driven by blink — impulse trigger, pulse/ripple across surface

Other mappings discussed:
- Alpha → material emission intensity (calm = glow)
- Beta → hue shift (tension = color change)
- Theta → blendshape inflate/organic deform (flow = expansion)
- Attention → texture scroll speed (focus = faster movement)
- Band_energy → overall scale or brightness
- Alpha_beta_ratio → color temperature (warm/cool)
- Theta_alpha_ratio → noise/distortion parameter

Resonite note: no custom shaders yet (on roadmap). Achieve effects through shader combinations, ProtoFlux-driven material parameters, and material stacking. `DynamicBlendShapeDriver` component for blendshapes, `Value Field Drive<float>` for material fields.

**OSC gotcha:** Resonite takes ownership of the OSC port — don't run anything else on port 9001. Running Resonite and the Python script on the same machine → use 127.0.0.1.

**THE 5 FILES BUILT SO FAR:**

All in one folder, run with `python main.py`:

Looking back through everything, a few things stand out that we touched on but didn't fully resolve or build:

**The Arduino sketch itself**

We never actually wrote or verified the Arduino sketch. We assumed it's outputting the standard Brain Library CSV format because your BrainGrapher already works, but it's worth confirming — specifically whether your sketch outputs the blink field as a 12th column or not, since the parser handles both but the blink value being zero forever would be a silent failure. Worth adding to the context doc.

**The 9600 vs 57600 baud decision has a real consequence**

At 9600 baud you get the processed band values once per second. At 57600 baud with the resistor mod to the TGAM board you get raw EEG at 512Hz — a completely different data stream that would need its own processing pipeline. The current code assumes 9600 and the processed Brain Library output. If you ever switch to 57600 raw mode the parser breaks entirely and you'd need a different approach. This should be clearly noted in the config so you don't accidentally switch baud rate in the UI and wonder why everything stops working.

**The 1 packet/second update rate and what it means for the skin**

We talked about EMA and Resonite's SmoothLerp hiding the update rate, but didn't fully think through what this means aesthetically. At 1Hz the skin can't respond faster than once per second at the source. For something that's supposed to feel like breathing chromatophores, you want the *visual* motion to be continuous even when the data is sparse. The recommendation we landed on — drive animated texture parameters and noise offsets that have their own motion, with EEG steering the character of that motion rather than being the sole source — is the right approach, but it means the Resonite setup needs to be built with this in mind from the start. The skin should have baseline animation that EEG modulates, not EEG being the only thing moving it.

**Port persistence across machines**

The config has `SERIAL_PORT = ""` for auto-detect, but the UI also lets you select a port manually. What the code doesn't do is remember your last selection between sessions. On a different machine COM3 might be something else entirely. This is minor but worth a note — the UI currently resets to blank every launch, and if the Arduino auto-detection works reliably enough this might not matter, but if you're switching between your stream machine and a laptop regularly it's a small friction point.

**The Quest 3 and Air Link latency**

We discussed the hardware setup but never addressed the fact that Quest 3 running Resonite via Air Link or Virtual Desktop introduces its own latency on top of the 1Hz EEG update rate. The OSC packets go from Python on the PC to Resonite on the PC (localhost), so the OSC path itself is zero latency — but the visual result you see in the headset has Air Link's rendering pipeline on top. This probably doesn't matter for slow expressive changes but is worth knowing if something feels sluggish.

**No graceful handling of the TGAM "poor signal" behavior**

The TGAM has a specific behavior documented in the protocol: if it receives poor signal for 7 consecutive seconds it resets, and if the electrodes are off-head for 4 consecutive seconds it resets. After a reset the attention/meditation values stop updating until contact is reestablished. The current code gates on signal quality correctly, but doesn't specifically handle the reset state — there's a difference between "signal quality is 50" (marginal contact) and "signal quality is 200 and has been for 7 seconds" (board has reset and needs a moment to re-acquire). In practice this probably just looks like a longer freeze which is fine, but worth knowing if you see the system take longer to recover than expected after taking the headset off.

**The design doc doesn't reference the code**

The checkpoint design doc we wrote earlier describes the planned architecture. Now that the code exists, the doc is slightly out of date — for example it describes a `serial_reader.py` as a separate file but the final implementation folded that into `main.py`. Minor, but if you're handing this to someone else or coming back to it after a break the doc and code should agree.

**What we never discussed: how other people in the space will actually perceive the skin**

This is more conceptual than technical but it's the whole point of the project. In Resonite, other users see your avatar. If your skin is reacting to your brain state in real time, other people are seeing a continuous readout of your internal state — which is exactly the post-symbolic communication Lanier was describing. But that means the mapping choices you make in Resonite are actually a design language. What does high beta look like? What does deep meditation look like? These are aesthetic and communicative decisions, not just parameter tweaks. It might be worth sketching out what emotional/cognitive states you want to be legible to others vs what you want to keep subtle. The octopus doesn't just react randomly — the patterns mean something to other octopi. That intentional design layer is the part nobody in the BCI/VR community has really worked out yet, and it's where your project could be genuinely new.

#include <SoftwareSerial.h>
#include <Brain.h>

// Set up SoftwareSerial to listen on Pin 10
// Pin 10 = RX (Connect headset "T" wire here)
// Pin 11 = TX (Unused, but required by the library)
SoftwareSerial softSerial(10, 11);

// Tell the Brain library to use our SoftwareSerial instead of the default
Brain brain(softSerial);

void setup() {
    // Hardware Serial for the computer (USB)
    Serial.begin(9600);
    
    // Software Serial for the headset
    softSerial.begin(9600);
    
    Serial.println("--- Bridge Starting: Listening on Pin 10 ---");
}

void loop() {
    // Check if a full packet has arrived from the headset
    if (brain.update()) {
        // This sends: signal, attention, meditation, delta, theta, etc.
        // This is exactly what your main.py / processor.py is looking for.
        Serial.println(brain.readCSV());
    }
}
